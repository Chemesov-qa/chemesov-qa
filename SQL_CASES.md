# 🗄 Tестирования базы данных SQL

В рамках проекта «Кордон» выполнял прямую проверку корректности сохранения данных в PostgreSQL.<br>
Ниже - примеры SQL-запросов, которые применялись для контроля целостности информации на уровне базы.

---

### Кейс №1: Верификация записи события
**Задача:** Убедиться, что после физического проезда автомобиля через зону контроля, запись в БД создается корректно и содержит все необходимые атрибуты (номер, время, ID камеры).

*   **Запрос:**
    ```sql
    SELECT 
    id AS "ID записи", 
    plate_number AS "Госномер", 
    event_time AS "Время проезда", 
    camera_id AS "ID камеры", 
    confidence_score AS "Точность распознавания"
    FROM recognition_events
    WHERE plate_number = 'A123BC777' 
    AND event_time >= '2024-05-20 10:00:00'
    AND confidence_score > 0.8
    ORDER BY "Время проезда" DESC;
    ```
*   **Что проверяем:** Соответствие Timestamp в базе реальному времени проезда и наличие высокого порога уверенности распознавания (не ниже 0.8).
*   **Ожидаемый результат:**<br>
>Запрос возвращает ровно 1 строку с confidence_score > 0.8.<br>
>Если 0 строк — событие не сохранилось.<br>
>Если больше 1 — есть дублирование.<br>

### Кейс №2: Проверка связей между таблицами
**Задача:** Убедиться, что события в базе данных корректно привязаны к существующим устройствам и их гео-позициям.

*   **Запрос:**
    ```sql
    SELECT 
    e.event_time AS "Время события", 
    e.plate_number AS "Госномер", 
    d.device_name AS "Имя устройства", 
    d.location_address AS "Адрес установки"
    FROM recognition_events AS e
    JOIN devices AS d ON e.device_id = d.id
    WHERE d.status = 'Online'
    ORDER BY "Время события" DESC
    LIMIT 10;
    ```
*   **Что проверяем:** Корректность связей между событиями и устройствами. Убеждаемся, что каждое событие привязано к существующей и работающей камере (статус Online), а адрес установки отображается без ошибок.

### Кейс №3: Поиск логических ошибок
**Задача:** Проверить систему на наличие бага «Размножение дупла» — ситуации, когда из-за нестабильного сетевого сигнала один проезд фиксируется в базе дважды.

*   **Запрос:**
    ```sql
    SELECT 
    plate_number AS "Госномер", 
    event_time AS "Время проезда", 
    device_id AS "ID устройства", 
    COUNT(*) AS "Количество дублей"
    FROM recognition_events
    GROUP BY "Госномер", "Время проезда", "ID устройства"
    HAVING COUNT(*) > 1;
    ```
*   **Что проверяем:** Уникальность каждой фиксации. Если запрос возвращает строки — это свидетельствует о дефекте в логике сохранения данных (Race Condition или отсутствие Unique Constraint).

### Кейс №4: Проверка производительности
**Задача:** Быстрая сверка статистики за час для сравнения данных в UI-отчете и в «сырой» базе.

*   **Запрос:**
    ```sql
    SELECT 
    camera_id AS "ID камеры", 
    COUNT(*) AS "Всего автомобилей за час"
    FROM recognition_events
    WHERE event_time BETWEEN '2024-05-20 09:00:00' AND '2024-05-20 10:00:00'
    GROUP BY "ID камеры";
    ```
*   **Что проверяем:** Сходятся ли цифры в графиках мониторинга с реальным количеством записей в таблице.

### Кейс №5: Проверка полноты критических данных
**Задача:** Найти записи, в которых отсутствуют данные, обязательные для выписки постановления.

*   **Запрос:**
    ```sql
    SELECT 
    id AS "ID Записи", 
    plate_number AS "Номер", 
    speed AS "Скорость", 
    lane_id AS "Полоса"
    FROM recognition_events
    WHERE plate_number IS NULL OR speed IS NULL OR lane_id IS NULL;
    ```
*   **Что проверяем:** Корректность работы фильтров и отсутствие «битых» записей в БД.

### Кейс №6: Аудит безопасности и прав доступа
**Задача:** Убедиться, что записи создаются только авторизованными системными агентами.

*   **Запрос:**
    ```sql
    SELECT 
    id AS "ID события", 
    created_by AS "Кем создана запись"
    FROM recognition_events
    WHERE created_by NOT IN ('camera_agent', 'system_worker');
    ```
*   **Что проверяем:** Отсутствие следов ручного вмешательства или несанкционированных правок в таблице событий.

### Кейс №7: Проверка логики фиксации превышений
**Задача:** Убедиться, что система не маркирует автомобили как нарушители, если скорость находится в пределах погрешности измерения (+2 км/ч к лимиту).

*   **Запрос:**
    ```sql
    SELECT 
    id AS "ID записи", 
    speed AS "Зафиксированная скорость", 
    speed_limit AS "Лимит на участке"
    FROM recognition_events
    WHERE speed <= speed_limit + 2  -- Порог погрешности
    AND violation_type = 'Speeding';
    ```
*   **Что проверяем:** Отсутствие ложноположительных срабатываний на граничных значениях скорости.

---

### Кейс №8: Проверка гео-координат
**Задача:** Убедиться, что камера передает корректные GPS-координаты и они не "нулевые".

*   **Запрос:**
    ```sql
    SELECT 
    camera_id AS "ID Камеры", 
    latitude AS "Широта", 
    longitude AS "Долгота"
    FROM recognition_events
    WHERE latitude = 0 OR longitude = 0 OR latitude IS NULL;
    ```
*   **Что проверяем:** Исправность GPS-модуля на аппаратном комплексе.

### Кейс №9: Проверка дублирования IP-адресов
**Задача:** Выявить конфликт IP-адресов в сетевой инфраструктуре «Кордона».

*   **Запрос:**
    ```sql
    SELECT 
    ip_address AS "IP-адрес", 
    COUNT(*) AS "Кол-во устройств"
    FROM cameras_registry
    GROUP BY ip_address
    HAVING COUNT(*) > 1;
    ```
*   **Что проверяем:** Отсутствие сетевых коллизий при конфигурации оборудования.

---

### Кейс №10: Сверка часовых поясов оборудования
**Задача:** Проверить, нет ли расхождения между временем на камере и временем сервера.
*   **Запрос:**
    ```sql
    SELECT 
    d.device_name AS "Камера", 
    e.event_time AS "Время события", 
    e.received_at AS "Время получения сервером",
    (e.received_at - e.event_time) AS "Разница (сек)"
    FROM recognition_events AS e
    JOIN devices AS d ON e.device_id = d.id
    WHERE ABS(EXTRACT(EPOCH FROM (e.received_at - e.event_time))) > 5;
    ```
*   **Что проверяем:** Отсутствие рассинхрона между временем на устройстве и временем на сервере (разница `received_at` и `event_time` не должна превышать допустимые значения).
