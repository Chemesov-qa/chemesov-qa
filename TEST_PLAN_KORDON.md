# Тест-план: Приемочное тестирование обновления ПО v.3.6 для ПАК «Кордон»

**Документ:** TP-KORDON-3.6-RC1  
**Версия:** 1.0  
**Автор:** Инженер связи / Leonid  
**Дата:** 18.04.2026

## 1. Объект тестирования и объем
Объектом тестирования является процедура обновления программного обеспечения до версии **3.6** и проверка стабильности работы основной функциональности на группе из **4 (четырех)** тестовых комплексов ПАК «Кордон» перед массовым развертыванием (деплоем).

*   **Входит в тестирование:**
    *   Процесс установки обновления (скрипт деплоя) на целевую архитектуру.
    *   Обратная совместимость формата данных (JSON-пакеты) с существующим сервером ЦОД.
    *   Функционирование ключевых модулей **после** обновления (Распознавание ГРЗ, измерение скорости, запись на SD).
    *   Проверка механизма отката на случай неудачного обновления.
    *   Валидация новых параметров конфигурации из `changelog` версии 3.6 (согласно внутреннему списку изменений).
*   **Не входит в тестирование:**
    *   Полный цикл регресса всей системы «Кордон-М».
    *   Исправление багов, обнаруженных **до** установки версии 3.6 (известные проблемы v.3.5.x).

## 2. Окружение (Test Environment)
**Конфигурация тестовой группы:**

*   **Железо:** 4 х Камеры «Кордон-М» (Сн: KRD-101-11, KRD-101-12, KRD-101-13, KRD-101-14). 
    *   *Примечание:* Комплексы разнесены по разным сегментам сети для имитации реальных условий.
*   **ПО до обновления:** Версия прошивки 3.5.8 (Stable).
*   **Целевое ПО:** Образ обновления `kordon-update-3.6.0.bin`.
*   **Серверная часть:** Тестовый контур ЦОД, Docker-контейнер `collector:test`.
*   **Инструменты контроля:**
    *   `SSH-консоль` - для заливки прошивки и чтения локальных логов камеры.
    *   `Postman` - валидация API `/event` и `/status`.
    *   `Grafana` (дашборд `device_health`) - мониторинг загрузки CPU и температуры чипа во время и после апдейта.
    *   `GitLab CI Runner` - скрипт автоматизированной проверки версий.

## 3. Критерии тестирования
### 3.1. Критерии начала (Entry Criteria)
1. [x] Выпущен официальный релизный билд `v3.6.0` с подписью разработчика.
2. [x] На 4-х тестовых камерах остановлен поток боевых данных (переведены в режим "Тест/Отладка").
3. [x] Сделан полный бэкап конфигурационных файлов (`config.json` и `calibration.cfg`) с каждого комплекса.
4. [x] Утилита проверки целостности файловой системы не выявила ошибок на SD-картах.

### 3.2. Критерии завершения (Exit Criteria)
1. Успешное обновление ПО до версии 3.6 на всех 4-х устройствах без сбоев питания и "кирпичей".
2. **Нулевая толерантность к регрессу:** Функционал распознавания ГРЗ (Day/Night) и трекинг скорости работают не хуже, чем на версии 3.5.8.
3. Параметр `uptime` после апдейта составляет не менее **12 часов** без перезагрузок по `Watchdog`.
4. Подписан акт приемки тестовой группой (Инженер + Руководитель отдела эксплуатации).

## 4. Стратегия тестирования (План работ)
### Этап 1: Подготовка и установка (Pre-deploy)
*   **Действие:** Ручная загрузка прошивки через Web-интерфейс на **Один** комплекс (KRD-101-11).
*   **Проверка:** Сравнение контрольной суммы MD5 прошивки до и после загрузки.

### Этап 2: Функциональный минимум (Smoke Test)
*   **Объект:** KRD-101.
*   **Сценарий:** Проезд тестового авто (4 проезда: день, ночь, грязный номер, скорость +19 км/ч).
*   **Критерий успеха:** JSON-пакет с событиями доходит до тестового сервера за < 2 секунд. Координаты GPS фиксируются корректно.

### Этап 3: Масштабирование на группу (Rollout to 4 devices)
*   **Действие:** Параллельное обновление оставшихся 3-х комплексов (KRD-102-22, 103-33, 104-44) на каналах LTE.
*   **Фокус тестирования:** Проверка механизма докачки пакетов, если связь пропадала в момент перезагрузки устройства.

### Этап 4: Стабилизация и мониторинг
*   **Длительность:** 24 часа.
*   **Метрики контроля:**
    *   Объем свободной RAM (Memory Leak detection).
    *   Количество потерянных пакетов (Packet Loss) на интерфейсе `eth0` / `wwan0`.
    *   Ошибки сегментации (`Segmentation fault`) в `syslog`.

## 5. Риски и минимизация

| Риск | Влияние | План минимизации / Реакция |
| :--- | :--- | :--- |
| **Сбой питания во время прошивки** | Критическое<br> (Выход из строя загрузчика, «кирпич») | Проведение работ только при наличии ИБП на питающей линии.<br> Проверка состояния батареи ИБП перед стартом. |
| **Увеличение потребления трафика в v3.6** | Среднее | Мониторинг графика трафика на роутерах<br> в первые часы после апдейта. |
| **Несовместимость конфига** | Высокое | **Rollback Plan:** Разработан скрипт отката на `v3.5.8`<br> одной командой через SSH. |
| **Деградация качества ночной съемки** | Среднее | Валидация кадров с ИК-подсветкой.<br> В случае ухудшения - фиксация бага и **Block** релиза. |
