# Тест-план: Система автоматической фиксации нарушений ПДД «Кордон-М»

Этот тест-план описывает стратегию и объем тестирования программно-аппаратного комплекса (ПАК) мониторинга дорожного движения.

## 1. Объект тестирования (Scope)
Объектом тестирования является интеграция камеры «Кордон-М» с сервером мониторинга и корректность передачи данных о нарушениях.

*   **Входит в тестирование:**
    *   Процесс распознавания ГРЗ (государственных регистрационных знаков).
    *   Передача пакета данных (фото + JSON с метаданными: скорость, время, координаты).
    *   Работа системы в условиях обрыва связи (буферизация).
*   **Не входит в тестирование:**
    *   Физическая устойчивость кронштейнов крепления.
    *   Работа внутренних алгоритмов нейросети (Black Box тестирование).

## 2. Окружение (Test Environment)
*   **Железо:** Камера «Кордон-М», промышленный роутер Teltonika (4G/LTE).
*   **ПО сервера:** Ubuntu Server 22.04 LTS, PostgreSQL.
*   **Инструменты:** 
    *   `Postman` — для проверки API-запросов от камеры к серверу.
    *   `Wireshark` — для анализа TCP/IP пакетов.
    *   `Zabbix` — для мониторинга аптайма устройства.

## 3. Критерии тестирования
### 3.1. Критерии начала (Entry Criteria)
1. Установлена стабильная версия прошивки камеры.
2. Настроен VPN-туннель до тестового сервера.
3. Подготовлены тестовые ГРЗ (пластины для тестов).

### 3.2. Критерии завершения (Exit Criteria)
1. Выполнены все запланированные тест-кейсы (100%).
2. Отсутствуют открытые дефекты с приоритетом **High** и **Critical**.
3. Подтверждена целостность данных в БД после 24 часов непрерывной работы.

## 4. Стратегия тестирования
*   **Функциональное:** Проверка корректного определения скорости (± 2 км/ч).
*   **Нагрузочное:** Имитация плотного потока машин (3000 авто/час).
*   **Негативное:** Попытка передачи данных при низком уровне сигнала (-100 dBm).

## 5. Риски и минимизация

| Риск | Влияние | План минимизации |
| :--- | :--- | :--- |
| Потеря пакетов при переключении вышек сотовой связи | Высокое | Тестирование механизма доотправки (Retry Policy) из локального буфера. |
| Засветка номера ИК-подсветкой в ночное время | Среднее | Настройка динамической экспозиции в зависимости от освещенности. |
| Переполнение SD-карты памяти на камере | Высокое | Проверка скрипта автоочистки старых логов и архивов. |

## 6. График (Schedule)
*   Подготовка тестовых данных: 1 день.
*   Проведение тестов: 3 дня.
*   Анализ логов и фиксация багов: 1 день.

# Тест-план: Приемочное тестирование обновления ПО v.3.6 для ПАК «Кордон»

**Документ:** TP-KORDON-3.6-RC1  
**Версия:** 1.0  
**Автор:** Инженер связи / Leonid  
**Дата:** 18.04.2026

## 1. Объект тестирования (Scope)
Объектом тестирования является процедура обновления программного обеспечения до версии **3.6** и проверка стабильности работы основной функциональности на группе из **4 (четырех)** тестовых комплексов ПАК «Кордон» перед массовым развертыванием (деплоем).

*   **Входит в тестирование:**
    *   Процесс установки обновления (скрипт деплоя) на целевую архитектуру.
    *   Обратная совместимость формата данных (JSON-пакеты) с существующим сервером ЦОД.
    *   Функционирование ключевых модулей **после** обновления (Распознавание ГРЗ, измерение скорости, запись на SD).
    *   Проверка механизма отката (Rollback) на случай неудачного обновления.
    *   Валидация новых параметров конфигурации из `changelog` версии 3.6 (согласно внутреннему списку изменений).
*   **Не входит в тестирование:**
    *   Полный цикл регресса всей системы «Кордон-М».
    *   Исправление багов, обнаруженных **до** установки версии 3.6 (известные проблемы v.3.5.x).
    *   Тестирование производительности новой нейросети (рассматривается только стабильность инференса без деградации).

## 2. Окружение (Test Environment)
**Конфигурация тестовой группы (Canary Group):**

*   **Железо:** 4 х Камера «Кордон-М» (Сн: KRD-2101, KRD-2102, KRD-2103, KRD-2104).
    *   *Примечание:* Комплексы разнесены по разным сегментам сети для имитации реальных условий (1 - оптический линк, 3 - LTE/4G).
*   **ПО до обновления:** Версия прошивки 3.5.8 (Stable).
*   **Целевое ПО:** Образ обновления `kordon-update-3.6.0.bin`.
*   **Серверная часть:** Тестовый контур ЦОД, Docker-контейнер `collector:test`.
*   **Инструменты контроля:**
    *   `SSH-консоль` — для заливки прошивки и чтения локальных логов камеры.
    *   `Postman` — валидация API `/event` и `/status`.
    *   `Grafana` (дашборд `device_health`) — мониторинг загрузки CPU и температуры чипа во время и после апдейта.
    *   `GitLab CI Runner` — скрипт автоматизированной проверки версий (Healthcheck).

## 3. Критерии тестирования
### 3.1. Критерии начала (Entry Criteria)
1. [x] Выпущен официальный релизный билд `v3.6.0` с подписью разработчика.
2. [x] На 4-х тестовых камерах остановлен поток боевых данных (переведены в режим "Тест/Отладка").
3. [x] Сделан полный бэкап конфигурационных файлов (`config.json` и `calibration.cfg`) с каждого комплекса.
4. [x] Утилита проверки целостности файловой системы не выявила ошибок на SD-картах.

### 3.2. Критерии завершения (Exit Criteria)
1. Успешное обновление ПО до версии 3.6 на всех 4-х устройствах без сбоев питания и "кирпичей".
2. **Нулевая толерантность к регрессу:** Функционал распознавания ГРЗ (Day/Night) и трекинг скорости работают не хуже, чем на версии 3.5.8.
3. Параметр `uptime` после апдейта составляет не менее **12 часов** без перезагрузок по `Watchdog`.
4. Подписан акт приемки тестовой группой (Инженер + Руководитель отдела эксплуатации).

## 4. Стратегия тестирования (План работ)
### Этап 1: Подготовка и установка (Pre-deploy)
*   **Действие:** Ручная загрузка прошивки через Web-интерфейс на **Один** комплекс (KRD-2101) по оптике.
*   **Проверка:** Сравнение контрольной суммы MD5 прошивки до и после загрузки.

### Этап 2: Функциональный минимум (Smoke Test)
*   **Объект:** KRD-2101.
*   **Сценарий:** Проезд тестового авто (4 проезда: день, ночь, грязный номер, скорость +19 км/ч).
*   **Критерий успеха:** JSON-пакет с событиями доходит до тестового сервера за < 2 секунд. Координаты GPS фиксируются корректно.

### Этап 3: Масштабирование на группу (Rollout to 4 devices)
*   **Действие:** Параллельное обновление оставшихся 3-х комплексов (KRD-2102, 2103, 2104) на каналах LTE.
*   **Фокус тестирования:** Проверка механизма докачки пакетов, если связь пропадала в момент перезагрузки устройства.

### Этап 4: Стабилизация и мониторинг
*   **Длительность:** 24 часа.
*   **Метрики контроля:**
    *   Объем свободной RAM (Memory Leak detection).
    *   Количество потерянных пакетов (Packet Loss) на интерфейсе `eth0` / `wwan0`.
    *   Ошибки сегментации (`Segmentation fault`) в `syslog`.

## 5. Риски и минимизация

| Риск | Влияние | План минимизации / Реакция |
| :--- | :--- | :--- |
| **Сбой питания во время прошивки** | Критическое (Выход из строя загрузчика, «кирпич») | Проведение работ только при наличии ИБП на питающей линии. Проверка состояния батареи ИБП перед стартом. |
| **Увеличение потребления трафика в v3.6** | Среднее | Мониторинг графика трафика на роутерах Teltonika в первые часы после апдейта. |
| **Несовместимость конфига** | Высокое | **Rollback Plan:** Разработан скрипт отката на `v3.5.8` одной командой через SSH. |
| **Деградация качества ночной съемки** | Среднее | Валидация кадров с ИК-подсветкой. В случае ухудшения - фиксация бага и **Block** релиза. |
